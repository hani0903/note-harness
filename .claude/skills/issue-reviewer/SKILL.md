name: issue-reviewer
description: >
  이슈 문서(issues.md) 또는 GitHub issue를 리뷰할 때 사용하는 검토 스킬.

  사용자가 다음과 같은 요청을 하면 반드시 사용한다.

  * "이슈 리뷰해줘"
  * "이슈 분해가 잘 되었는지 확인해줘"
  * "수직 슬라이싱이 맞는지 봐줘"
  * "Acceptance Criteria 검토해줘"
  * "이슈 품질 확인해줘"
  * "이슈 작성했는데 피드백해줘"

  사용자가 명시적으로 스킬 이름을 언급하지 않아도
  이슈 검토 맥락이 감지되면 적극적으로 사용한다.

  이 스킬은 새로운 이슈를 생성하거나 수정하지 않는다.
  피드백은 체크리스트 기반으로 판단하며, 근거 없는 취향성 피드백은 제공하지 않는다.

  역할:
  * 이슈 형식 준수 여부 확인
  * 수직 슬라이싱 여부 검증
  * Acceptance Criteria의 테스트 가능성 검증
  * Ubiquitous Language 일관성 확인
  * 의존성 방향 검증 (개별 + 전체 시퀀스)
  * 이슈 크기 적절성 검증
  * 모호한 표현 탐지

  출력 형식:
  리뷰 대상 이슈를 항목별로 검토한 뒤 반드시 다음 3개 섹션으로 구성한다.

  * 🔴 Critical: 반드시 수정이 필요한 문제 (근거와 수정 방향 포함)
  * 🟡 Warning: 수정 권장 사항 (근거 포함)
  * ✅ Passed: 기준을 충족하는 항목

  이슈가 여러 개인 경우 이슈별로 검토한 뒤, 마지막에 시퀀스 전체 의존성을 별도 검토한다.

# Issue Review Checklist

체크리스트의 각 항목은 심각도 레이블을 가진다.
[C] = Critical (미충족 시 이슈 실행 불가 수준), [W] = Warning (품질 저하 위험)

---

## 1. 이슈 형식 준수

- [C] 설명 섹션이 있고, 사용자 가치 중심으로 1~3문장으로 작성되었는가?
- [C] 구현 범위 섹션이 있고, 수정 대상 파일/레이어가 명시되어 있는가?
- [C] 완료 조건 섹션이 있고, AC가 Given-When-Then 형식으로 작성되었는가?
- [W] 각 AC에 `(범위: 단위)` 또는 `(범위: 통합)` 레이블이 명시되어 있는가?

---

## 2. 수직 슬라이싱

판단 기준: "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?"

- [C] 이슈 단독으로 완료했을 때 관찰 가능한 동작이 존재하는가?
  - 실패 예: "API 레이어만 추가", "Context만 추가" — 이 상태로는 아무것도 동작하지 않음
- [C] 이슈가 레이어(API / 상태 / UI)를 수평으로 분리해서 나뉘어 있지 않은가?
- [W] 이 이슈만으로 TDD Red→Green→Refactor 사이클이 완결되는가?

---

## 3. Acceptance Criteria

**Given 검증**

- [C] Given에 추상 표현("데이터가 있을 때") 대신 실제 데이터 구조나 값이 명시되어 있는가?
  - 허용: `[{id:1, title:"React"}]`, `count === 0`, `userId: "abc"`
  - 불허: "노트가 있을 때", "사용자가 로그인된 상태", "리스트가 비어있을 때"
- [W] 경계 조건(빈 값, 최대값, 중복)이 별도 AC로 커버되어 있는가?

**When 검증**

- [C] When이 외부에서 관찰 가능한 행동인가? (함수 호출, 버튼 클릭, 이벤트 등)
  - 불허: "시스템이 처리할 때", "내부 상태가 변경될 때"

**Then 검증**

- [C] Then에 검증 가능한 값·개수·문구가 명시되어 있는가?
  - 허용: `filteredItems.length === 1`, `toast("저장되었습니다")가 표시됨`
  - 불허: "필터링된다", "업데이트된다", "저장된다"
- [C] 아래 금지어가 Then에 사용되고 있지 않은가?
  - `정상적으로` / `적절히` / `올바르게` / `보여진다` / `처리된다` / `업데이트된다` / `반영된다`
  - 금지 이유: 결과값 없이 결과를 서술하므로 테스트 코드로 1:1 변환 불가능

**범위 레이블 검증**

- [W] `(범위: 단위)` 레이블이 붙은 AC는 `renderHook` 또는 순수 함수 테스트로 검증 가능한가?
- [W] `(범위: 통합)` 레이블이 붙은 AC는 `render` + 사용자 인터랙션으로 검증 가능한가?

---

## 4. Ubiquitous Language

- [W] 이슈에서 사용된 핵심 용어가 prd.md / spec-fixed.md의 용어 정의와 일치하는가?
  - 예: prd.md에서 "노트"로 정의했는데 이슈에서 "메모"를 혼용하면 경고
- [W] 같은 이슈 내에서 동일 개념을 다른 단어로 표현하는 경우가 없는가?

_(prd.md나 spec-fixed.md가 제공되지 않은 경우, 이슈 내부 일관성만 검토하고 이 사실을 명시한다.)_

---

## 5. 이슈 크기

- [C] 반나절~하루 안에 Red→Green→Refactor 사이클이 완료 가능한 크기인가?
- [W] 구현 범위에 Hook / UI / API 변경이 3개 이상 포함된 경우, 분리 가능성을 검토했는가?
- [W] AC가 5개를 초과하는 경우, 이슈를 분리할 여지가 있는가?

---

## 6. 의존성 (이슈 단독)

- [C] 이 이슈가 의존하는 이슈가 먼저 완료되어야만 시작 가능한가? 순서가 올바른가?
- [C] 역방향 의존성이 없는가? (나중 이슈의 결과물을 이 이슈에서 사용하고 있지 않은가?)

---

## 7. 시퀀스 전체 의존성 (이슈 목록 전체 검토 시)

- [C] 전체 이슈 시퀀스가 "앞 이슈 결과 → 다음 이슈 입력" 체인을 형성하는가?
- [C] 중간 이슈가 빠져서 다음 이슈 구현이 불가능한 구간이 없는가?
- [W] PRD의 Out of Scope 항목이 어떤 이슈에도 포함되어 있지 않은가?

---

## 출력 형식 예시

```
## 리뷰 결과 — Issue 3: [필터] 태그 필터링

### 🔴 Critical
- **AC-1 Then 금지어**: "필터링된다"는 결과값 없는 서술.
  → `filteredItems.length === 1` 형태로 수정 필요.
- **수직 슬라이싱 위반**: Context 변경만 포함되어 있고 UI 반영이 없음.
  이슈 단독으로 동작이 관찰되지 않음.

### 🟡 Warning
- **AC 범위 레이블 누락**: AC-2에 (범위: 단위/통합) 레이블이 없음.
- **Given 추상적**: "태그가 선택된 상태"보다 `selectedTags: ["React"]`처럼 구체화 권장.

### ✅ Passed
- 이슈 형식(설명/구현 범위/완료 조건) 모두 존재.
- 의존성 순서 올바름 (Issue 2 → Issue 3).
- 하루 이내 구현 가능한 크기로 판단됨.
```
