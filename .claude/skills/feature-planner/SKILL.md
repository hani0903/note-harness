  맥락이 감지되면 적극적으로 사용한다.

# Feature Planner

아이디어를 실행 가능한 GitHub Issues로 변환하는 3단계 파이프라인이다.

```
spec-original.md (초기 아이디어)
    ↓ 단계 1 — 요구사항 인터뷰
spec-fixed.md (확정된 요구사항)
    ↓ 단계 2 — PRD + 기술 결정(ADR)
prd.md (단일 기준 문서)
    ↓ 단계 3 — 이슈 분해
issues.md + GitHub Issues
```

각 단계 끝에 `[GATE]`가 있다. 게이트는 사용자 확인 없이 절대 넘어가지 않는다.
AI가 게이트를 건너뛰면 후속 단계 전체가 잘못된 전제 위에 쌓인다.

---

## 단계 1: 요구사항 인터뷰

### 목적

`spec-original.md`의 모호성을 제거하고 엣지 케이스·결정 사항을 확정한다.

### 진행 방법

**한 번에 하나씩** 질문한다. 여러 질문을 한꺼번에 던지면 사용자가 압도되고 중요한 답변이 묻힌다.

아래 8개 영역을 순서대로 다루되, 이미 spec에 명확히 답이 있으면 건너뛰어도 된다.
각 질문에는 **추천 방식과 이유**를 함께 제시해 사용자의 판단을 돕는다.

| #   | 질문 영역                                                     |
| --- | ------------------------------------------------------------- |
| 1   | 이 기능의 primary user는 누구인가?                            |
| 2   | 최소 동작 시나리오 3개는? (가장 기본적인 사용 흐름)           |
| 3   | 데이터 저장 방식 — 기존 구조와의 관계는?                      |
| 4   | 경계 조건 — 최대값, 빈 값, 중복, 동시성은?                    |
| 5   | 에러 처리 — 실패 시 사용자에게 어떻게 표시하는가?             |
| 6   | 기존 UI 패턴과의 일관성 — 어떤 컴포넌트를 재사용할 수 있는가? |
| 7   | 성능 제약 — 허용 응답 시간, 데이터 크기 제한은?               |
| 8   | 향후 확장 가능성 — 이 결정이 다음 feature에 영향을 주는가?    |

### Ubiquitous Language 확정

인터뷰 중 등장하는 핵심 용어를 고정한다.
코드·문서·대화에서 **같은 단어가 같은 것을 가리켜야** 한다.
나중에 `spec-fixed.md`의 "용어 정의" 섹션에 명시한다.

### 산출물

`docs/features/{name}/spec-fixed.md`

```
[GATE] 사용자가 spec-fixed.md를 읽고 "확정"을 말할 때까지 단계 2로 넘어가지 않는다.
```

## 단계 2: PRD + 기술 결정(ADR)

### 목적

요구사항·디자인·기술 결정·범위를 하나의 문서로 통합해 단일 기준점을 만든다.
"이 기능에 대해 궁금하면 `prd.md`만 보면 된다"는 상태가 목표다.

### 2-1. PRD 뼈대 작성

먼저 기술 결정 섹션을 `TODO`로 비워두고 나머지를 채운다.

PRD 구조:

- **개요** — 기능의 목적과 범위
- **사용자 스토리** — 사용자 관점의 흐름
- **기술 결정** — (다음 단계에서 채움)
- **Out of Scope** — 이번에 하지 않을 것
- **용어 정의** — spec-fixed.md의 Ubiquitous Language 동기화

### 2-2. 아키텍처 3안 제안 & 비교

최소 3가지 안을 제안하고, 아래 **7가지 고정 기준**으로 비교 표를 작성한다.

| #   | 기준                 | 설명                                           |
| --- | -------------------- | ---------------------------------------------- |
| 1   | 데이터 구조          | 어떤 형태로 데이터를 저장/표현하는가           |
| 2   | API 레이어 변경지점  | 기존 API에서 무엇이 바뀌는가                   |
| 3   | 상태관리 변경지점    | Context/Store에서 무엇이 추가되는가            |
| 4   | 핵심 동작            | 이 기능의 가장 중요한 동작이 어떻게 구현되는가 |
| 5   | 컴포넌트 구조        | 어떤 컴포넌트가 추가/수정되는가                |
| 6   | 기존 패턴과의 일관성 | 코드베이스의 기존 관습과 얼마나 맞는가         |
| 7   | 테스트 용이성        | 단위 테스트, E2E 테스트 작성이 쉬운가          |

```
[GATE] 사용자가 3안 중 하나를 선택할 때까지 대기한다.
```

### 2-3. 선택한 안을 ADR로 작성

PRD의 "기술 결정" 섹션을 아래 4요소로 채운다.
단점 없는 ADR은 신뢰할 수 없다 — Consequences에 반드시 트레이드오프를 명시한다.

```markdown
### [결정 제목]

**Context** — 이 결정이 필요한 맥락. 무엇을 풀고 있는가.
**Decision** — 무엇을 선택했는가.
**Alternatives** — 거부한 안과 각각의 거부 이유. (이유 없는 거부 금지)
**Consequences** — 트레이드오프. 장점뿐 아니라 단점도 반드시 명시.
```

### 2-4. Out of Scope 작성

"하지 않을 것"을 구체적으로 나열한다.
명시하지 않으면 AI가 맥락을 확장 해석해 범위 초과 구현을 한다.

```
[GATE] 사용자가 Out of Scope를 읽고 확정할 때까지 대기한다.
```

### 산출물

`docs/features/{name}/prd.md`

## 단계 3: 이슈 분해

### 목적

PRD를 독립적으로 TDD 사이클을 돌릴 수 있는 작업 단위로 변환한다.

### 수직 슬라이싱 원칙

> "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?"

- **Yes** → 수직 슬라이스. TDD 사이클을 돌릴 수 있다.
- **No** → 다른 이슈와 합치거나 다시 나눈다.

**수평 슬라이싱 금지.** 레이어별 분리(API 이슈 → Context 이슈 → UI 이슈)는
마지막 이슈까지 완료해야 동작이 나오므로 이슈 단위 TDD가 불가능하다.

**이슈 크기:** 반나절~하루 안에 Red→Green→Refactor 사이클 완료 가능한 것.

**의존성 순서:** 앞 이슈의 결과가 다음 이슈의 입력이 되도록 배치. 역방향 개발 금지.

### 이슈 형식

각 이슈를 아래 구조로 작성한다.

```markdown
## Issue N: [태그] 제목

### 설명

사용자 가치 중심으로 1~3문장.

### 구현 범위

이 이슈에서 건드리는 파일/레이어 목록.

### 완료 조건 (Acceptance Criteria)

- [ ] AC 항목

### 시나리오

**Given** [사전 조건]
**When** [행동]
**Then** [기대 결과]
```

시나리오는 Given-When-Then 형식으로 작성한다.
AC는 다음 단계의 테스트 시나리오 씨앗이 되므로 구체적으로 작성한다.

### GitHub 등록 (선택)

```bash
gh issue create --title "이슈 제목" --body "설명 + AC + 의존성"
gh project item-add <PROJECT_NUMBER> --owner <OWNER> --url <ISSUE_URL>
```

### 산출물

`docs/features/{name}/issues.md`

```
[GATE] 사용자가 이슈 목록을 읽고 수직 슬라이스 기준 충족을 확인할 때까지 대기한다.
```
