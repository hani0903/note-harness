# TDD Refactor

`tdd-green` 이 만들어 둔 **전부 초록(Green)인 상태**를 입력으로 받아, **동작을 바꾸지 않고 코드 구조만** 개선한다. Refactor 의 정의 그대로다 — _겉으로 보이는 행동은 그대로 두고 내부 구조를 정리한다._ 테스트 묶음이 "행동이 그대로임"을 증명하는 안전망이므로, 매 변경마다 테스트가 계속 초록인지 확인하며 나아간다.

이 스킬은 **이번 이슈에서 바뀐 `src/` 의 비테스트 코드**만 손댄다. 테스트 파일은 사양이자 안전망이라 건드리지 않고, 이슈 범위 밖 파일도 만지지 않는다.

# 입력

- `$ARGUMENTS` — GitHub 이슈 번호 (예: `6`)
- 슬래시 호출: `/tdd-refactor 6`
- 전제: `tdd-green` 이 끝나 해당 이슈의 테스트가 전부 통과한다.
- 이슈 번호의 쓰임: `docs/features/{feature}/issue-{N}.md` 를 읽어 **무엇을 만든 코드인지 맥락을 잡고**, 보고서에서 대상을 지칭하는 데 쓴다. 실제 **수정 범위는 이슈 문서가 아니라 `git diff main...HEAD` 결과로 정해진다** (아래 단계 3).

# 핵심 원칙

1. **행동 보존 (Behavior-preserving)**
   - 입력→출력, 렌더 결과, 호출 인자 등 **겉으로 관찰되는 동작은 한 톨도 바뀌면 안 된다.** 바뀌었다면 그건 리팩토링이 아니라 기능 변경이다.
   - 그래서 새 기능·새 분기·새 옵션을 추가하지 않는다. "이왕 정리하는 김에"는 Refactor 의 일이 아니다.
   - 안전망은 테스트다. 테스트가 계속 초록이면 동작이 보존됐다는 신호이고, 빨개지면 동작을 바꿨다는 신호다.

2. **한 번에 하나씩, 매번 검증**
   - 변경 1개 → `npm test` → 초록 확인 → 다음. 여러 개를 몰아서 바꾼 뒤 한꺼번에 돌리면, 무엇이 무엇을 깨뜨렸는지 분간할 수 없다.
   - 각 변경은 **되돌리기 쉬운 작은 단위**로 유지한다 (단계 6의 롤백이 깔끔해진다).

3. **테스트가 의존하는 표면은 얼지 않은 듯 얼어 있다**
   - 테스트 파일은 수정하지 않는데(아래 제약), 테스트는 대상 모듈의 **export 이름·시그니처·역할(role)·텍스트** 등을 import 하거나 단언한다.
   - 따라서 테스트가 직접 참조하는 공개 표면(export 함수명, 컴포넌트 Props, `getByRole`/`getByText` 가 보는 마크업의 의미)은 **사실상 변경 불가**다 — 바꾸면 테스트가 깨지는데 테스트를 고칠 수 없기 때문이다.
   - 반대로 **내부 구현**(지역 변수명, 함수 분리/합성, 중복 제거, 매직넘버 상수화 등)은 자유롭게 정리할 수 있다. 리팩토링의 주 무대는 여기다.

4. **승인 게이트**
   - 무엇을 왜 바꿀지 목록으로 먼저 보고하고, 사용자 승인 후에 손을 댄다 (단계 4). 리팩토링은 취향이 갈리는 영역이라, 합의 없이 코드를 갈면 되돌리는 비용이 크다.

5. **커밋은 사용자 몫**
   - 이 스킬은 작업 중 커밋하지 않는다. 성공한 변경은 워킹 트리에 누적되고, 마지막에 사용자가 직접 커밋한다 (`tdd-red`/`tdd-green` 과 동일한 흐름).

# 점검 기준 (리팩토링 후보를 찾는 렌즈)

대상 파일을 읽을 때 아래 다섯 렌즈로 본다. 모든 코드를 다 고치라는 게 아니라, **명확히 개선되는 곳만** 후보로 올린다.

- **중복 제거** — 같은 로직이 두 곳 이상에서 반복되는가? (단, 한 번만 더 나온다고 무조건 추출하지 않는다 — 단계 4에서 판단)
- **네이밍 명확성** — 변수/함수/상수 이름이 의도를 드러내는가? (예: `data`, `temp`, `flag` 처럼 모호한 이름. CLAUDE.md 의 네이밍 규약과 어긋나는 이름.)
- **단일 책임** — 한 함수/컴포넌트가 너무 많은 일을 떠안고 있어 쪼개면 읽기 쉬워지는가?
- **불필요한 복잡도** — 매직 넘버, 과도한 중첩(깊은 if/삼항), 한 번만 쓰는데 만들어 둔 추상화 등.
- **CLAUDE.md 컨벤션 불일치** — 프로젝트가 정한 패턴(네이밍 표, early return 분기, named export, 상태 관리 위치, 스타일링 규칙 등)과 어긋나는 코드.

> 이 다섯은 출발점이다. CLAUDE.md 에 그 프로젝트만의 "일관성 없는 패턴" 같은 안내가 있으면 그것도 후보의 근거가 된다.

# 파이프라인

```
tdd-green 완료 (이슈 테스트 전부 Green)
        │
        ▼ 1. CLAUDE.md 읽기 — 프로젝트 컨벤션 파악
        ▼ 2. npm test — 전체 통과부터 확인 (시작점이 초록인지)
        ▼ 3. git diff main...HEAD --name-only -- src/ → 테스트 파일 제외 → 점검
        ▼ 4. 리팩토링 대상 목록 보고 → 사용자 승인 대기  [GATE]
        │
        ├─▶ 5. 변경 1개 적용 → npm test → 초록 확인 ◀─┐
        │   6. 빨개지면 그 변경만 되돌리고 다른 접근 ──┘
        │
        ▼ 7. 전후 변경 요약
        ▼ (커밋은 사용자가 직접)
```

---

## 단계 1: CLAUDE.md 읽기

프로젝트 루트의 `CLAUDE.md` 를 읽어 컨벤션을 파악한다 — 네이밍 규약, 컴포넌트/상태 관리 패턴, export 방식, 스타일링 규칙, 그리고 "주의해야 할 일관성 없는 패턴" 같은 안내. 리팩토링의 방향("무엇이 이 프로젝트에서 좋은 코드인가")은 너의 일반론이 아니라 **이 문서**가 정한다.

## 단계 2: 시작점이 초록인지 확인

```bash
npm test
```

- 전부 통과해야 리팩토링을 시작한다. **시작점이 빨강이면 그건 Refactor 가 아니라 Green 단계의 미완성** — 멈추고 사용자에게 알린 뒤 `tdd-green` 으로 돌려보낸다.
- 이 초록 상태가 이후 모든 변경의 기준선(baseline)이다.

## 단계 3: 리팩토링 대상 식별

이번 이슈에서 바뀐 `src/` 파일만 본다.

```bash
git diff main...HEAD --name-only -- src/
```

- `main...HEAD`(세 점)는 현재 브랜치가 main 과 갈라진 지점 이후 바뀐 파일을 보여준다 — 즉 "이 이슈가 건드린 것".
- 결과에서 **`*.test.ts` / `*.test.tsx` 를 제외**한다 (테스트는 수정 대상이 아니다).
- 남은 비테스트 파일을 읽고, 위 **점검 기준** 다섯 렌즈로 후보를 모은다.
- 결과가 비어 있으면(이슈가 src 를 안 바꿨거나 브랜치/베이스가 예상과 다르면) 멈추고 사용자에게 상황을 보고한다.

## 단계 4: 대상 목록 보고 → 승인 [GATE]

찾은 후보를 **무엇을 / 왜 / 어떻게** 형식으로 묶어 보고한다. 각 항목은:

- **대상**: `파일:라인` 또는 함수/컴포넌트 이름
- **문제**: 다섯 렌즈 중 어디에 걸리는지 + CLAUDE.md 근거(있으면)
- **제안**: 어떻게 바꿀지 (행동 보존 범위 안에서)

그리고 **공개 표면을 건드리는 항목은 따로 표시**한다 — 테스트가 그 이름/역할을 참조하면 변경 불가일 수 있으므로(원칙 3), 그런 항목은 보통 후보에서 빼거나 "내부만" 으로 한정한다.

사용자가 승인(또는 일부만 선택)할 때까지 코드를 바꾸지 않는다. 근거 없는 취향성 변경(동작·가독성에 차이 없는 단순 스타일 선호)은 후보에 올리지 않는다 — 포맷은 어차피 pre-commit 훅이 처리한다.

## 단계 5: 변경 루프 (하나씩)

승인된 목록을 하나씩 처리한다.

1. **변경 전 상태를 기억한다.** 곧 되돌릴 수 있도록, 바꿀 코드 영역의 현재 내용을 정확히 파악해 둔다 (단계 6의 롤백 근거).
2. **변경 1개를 적용한다.** 한 후보 = 가능한 한 하나의 논리적 수정. 큰 후보면 더 작은 단계로 쪼갠다.
3. **테스트를 돌린다.**
   ```bash
   npm test
   ```

   - 빠르게 좁혀 보려면 `npm test -- <파일경로>` 로 해당 파일만 먼저 확인하되, **그 변경이 다른 모듈에 파급될 수 있으면(공유 모듈·타입·컨텍스트) 반드시 전체 `npm test`** 로 회귀를 본다.
4. **전부 초록이면** 그 변경을 유지하고 다음 후보로. 커밋은 하지 않는다(원칙 5).

## 단계 6: 깨지면 즉시 롤백

테스트가 하나라도 빨개지면:

- **방금 적용한 변경만 되돌린다** — 변경 전 코드로 복원한다(단계 5-1에서 기억해 둔 상태). 누적된 이전 성공들은 그대로 둔다.
  - 커밋을 안 하는 흐름이라 `git checkout -- <file>` 은 **쓰지 않는다** — 그 파일에 누적된 이전 성공까지 날아간다. 방금 만든 수정만 역으로 되돌린다.
  - 한 변경이 여러 군데를 건드렸다면 그 군데를 모두 원복한다. 이래서 변경을 작게 유지하는 게 중요하다(원칙 2).
- 되돌린 뒤 `npm test` 로 **다시 초록(기준선 복귀)** 임을 확인한다.
- 그 후보는 **다른 접근**으로 재시도하거나, 마땅한 방법이 없으면 건너뛰고 그 사실을 요약에 남긴다. 테스트가 깨지는 리팩토링을 억지로 밀어붙이지 않는다 — 깨진다는 건 그 변경이 동작을 바꿨다는 뜻이다.

## 단계 7: 전후 요약

마지막에 한 번에 보고한다.

- **적용한 리팩토링** — 항목별 전/후 요약 (무엇을 왜 바꿨는지, `파일:라인`).
- **건너뛴 후보** — 승인됐으나 테스트를 깨서(또는 다른 이유로) 보류한 것과 그 사유.
- **검증 상태** — 최종 `npm test` 가 전부 초록인지, (가능하면) `npx tsc --noEmit` 도 통과하는지.
- **남은 제안** — 이번 범위(이슈가 바꾼 파일) 밖이라 손대지 않았지만 눈에 띈 개선거리(있으면, 참고용으로만).
- 커밋은 사용자가 직접 한다는 안내.

## 제약 (반드시 지킬 것)

- **새 기능 추가 금지** — 동작이 바뀌면 리팩토링이 아니다. 관찰 가능한 행동(출력·렌더·호출 인자)은 그대로 둔다.
- **테스트 파일 수정 금지** — `*.test.ts(x)` 는 행동 보존을 증명하는 안전망이다. 리팩토링을 통과시키려고 테스트를 고치는 순간 안전망이 사라진다. 테스트가 막는다면 그건 동작을 바꾸려 한다는 신호다.
- **src/ 밖 수정 금지** — 설정·문서·빌드 파일 등은 이 단계의 대상이 아니다.
- **git diff 범위 밖 수정 금지** — `git diff main...HEAD -- src/` 에 없는 파일은 이번 이슈가 만든 게 아니므로 건드리지 않는다. 그 밖의 개선거리는 요약에 "남은 제안"으로만 남긴다.
- **승인 없이 변경 금지** — 단계 4 게이트를 통과한 항목만 손댄다.

## 리팩토링 예시 (행동 보존의 감각)

아래는 "동작은 그대로, 구조만" 이 어떤 모습인지 보여주는 예시다. 실제 후보는 항상 대상 코드와 CLAUDE.md 에서 나온다.

### 매직 넘버 → 의미 있는 상수 (내부 변경, 안전)

```tsx
// before
<textarea rows={14} ... />

// after
const EDITOR_ROWS = 14;
// ...
<textarea rows={EDITOR_ROWS} ... />
```

렌더 결과는 동일(`rows=14`) → 테스트 영향 없음. 의도가 이름으로 드러난다.

### 불리언 네이밍 컨벤션 정렬 (내부 변경, 안전)

CLAUDE.md 가 "불리언 상태는 `is` 접두어"를 규약으로 두고 `saving` 을 일관성 없는 예로 든 경우:

```tsx
// before
const [saving, setSaving] = useState(false);
// after
const [isSaving, setIsSaving] = useState(false);
```

이 상태가 **컴포넌트 내부에서만** 쓰이고 테스트가 직접 참조하지 않으면 안전하다. (테스트가 `saving` 을 import 하거나 단언한다면 → 공개 표면 → 변경 불가, 후보에서 제외.)

### 하면 안 되는 예 (행동 변경)

```tsx
// before: 빈 제목이면 조용히 무시
if (!title.trim()) return;

// after: ❌ 안내 메시지 추가 — 새 동작이다. 리팩토링이 아니다.
if (!title.trim()) {
  showToast('제목을 입력하세요');
  return;
}
```

## 자주 하는 실수 (피할 것)

- **"이왕 정리하는 김에" 기능을 슬쩍 추가**
  - 검증을 더 빡세게 하거나, 엣지 처리를 넣거나, 옵션을 추가하는 건 동작 변경이다. 그건 새 이슈/시나리오의 일이지 Refactor 가 아니다.

- **테스트가 깨지자 테스트를 고침**
  - 안전망을 잘라내는 행위다. 테스트가 깨졌다는 건 리팩토링이 동작을 바꿨다는 뜻이니, 코드를 되돌리고 접근을 바꾼다.

- **공개 표면을 무심코 rename**
  - 테스트가 import 하는 export 함수명이나 `getByRole`/`getByText` 가 보는 텍스트·역할을 바꾸면 테스트가 깨지는데, 테스트는 못 고친다. 내부 이름과 공개 표면을 구분한다(원칙 3).

- **여러 변경을 몰아서 적용**
  - 한꺼번에 바꾸고 `npm test` 한 번 돌리면, 깨졌을 때 원인을 못 짚고 롤백 범위도 커진다. 하나씩.

- **`git checkout -- <file>` 로 롤백**
  - 커밋을 안 하는 흐름에서는 누적된 이전 성공까지 날린다. 방금 만든 변경만 역으로 되돌린다(단계 6).

- **범위 밖까지 손대기**
  - git diff 에 없는 파일이나 src 밖을 건드리는 것. 좋은 개선거리라도 이번 이슈 범위가 아니면 "남은 제안"으로만 남긴다.

- **취향성 변경을 후보로 올림**
  - 동작에도 가독성에도 실질 차이가 없는 단순 스타일 선호는 노이즈다. 포맷·공백은 pre-commit 훅(Prettier/ESLint)이 알아서 한다.
