(참고: 이 스킬은 tsc+audit+시크릿 grep 기반의 가벼운 커밋 전 게이트다. 브랜치 전체 diff 를 깊게 읽는 종합 보안 리뷰가 필요하면 별도 도구를 쓴다.)

# Security Review (커밋 전 점검)

`tdd-refactor` 까지 끝난 코드를 **커밋하기 직전**, 두 가지를 빠르게 점검한다 — **타입이 안전한가**(빌드가 깨지지 않는가)와 **알려진 취약점·노출된 시크릿이 없는가**. 그리고 찾은 것을 **세 등급으로 분류**해, 무엇을 지금 고치고 무엇을 미뤄도 되는지 개발자가 한눈에 판단하게 한다.

핵심은 **분류**다. 스캔 도구(`tsc`, `npm audit`)는 "문제 있음/없음"만 쏟아낼 뿐, **무엇이 이 프로젝트에서 실제 위험인지**는 판단하지 못한다. 그 판단을 대신 해주고, 자동으로 고치는 건 위험이 명확한 것만, 그것도 승인 후에 한다.

# 입력

- `$ARGUMENTS` — GitHub 이슈 번호 (예: `6`)
- 슬래시 호출: `/security-review 6`
- 전제: 커밋하려는 변경이 워킹 트리에 있고, 테스트는 통과(Green) 상태다.
- 이슈 번호의 쓰임: 보고서에서 점검 대상을 지칭하고, 시크릿 grep 의 우선순위를 이번 이슈가 바꾼 파일(`git diff main...HEAD --name-only -- src/`)에 두는 데 쓴다. 단, `tsc`/`audit` 는 본래 프로젝트 전체를 본다.

# 핵심 원칙

1. **스캔은 기계가, 분류는 판단이 한다**
   - `tsc`·`npm audit` 결과를 그대로 "할 일"로 받지 않는다. 각 항목이 이 프로젝트(여기선 Vite 프런트엔드 + JSON Server)에서 **실제로 어떤 위험인지** 따져 등급을 매긴다.
   - 같은 "critical" 도 런타임 번들에 실리는 의존성이면 위험하고, 빌드/테스트 전용 devDependency 면 대개 그렇지 않다 (아래 `## 분류 기준`).

2. **자동 수정은 위험이 명확한 것만, 승인 후에만**
   - "즉시 수정 필요" 로 분류되고 개발자가 승인한 항목만 손댄다. "권장"·"무시"는 보고만 하고 건드리지 않는다.
   - 판단이 갈리는 항목(이 의존성을 올려도 되나? 이 타입 오류를 어떻게 고치나?)은 임의로 고치지 말고 개발자에게 묻는다.

3. **`npm audit fix --force` 금지**
   - `--force` 는 semver 호환을 무시하고 메이저 버전을 갈아끼워 **동작을 깨뜨릴 수 있다.** 커밋 전 점검 단계에서 그건 받아들일 수 없는 부작용이다.
   - 의존성 수정이 필요하면 `npm audit fix`(semver 호환 범위 내) 까지만 쓰거나, 그걸로 안 되면 개발자에게 보고하고 멈춘다.

4. **테스트 파일·범위 밖은 건드리지 않는다**
   - `*.test.ts(x)` 는 수정하지 않는다 (안전망).
   - 타입 오류를 고칠 때도 그 오류를 낸 코드만 고친다. "겸사겸사" 리팩토링하지 않는다 (그건 `tdd-refactor` 의 일).

5. **클린의 정의 = 즉시-수정 항목 0**
   - 마지막 재스캔에서 "즉시 수정 필요" 가 모두 해소됐는지를 본다. 남은 audit 취약점이 0 이어야 한다는 뜻이 **아니다** — "권장"·"무시" 로 분류해 의도적으로 남긴 것은 클린 상태와 모순되지 않는다.

# 분류 기준 (이 스킬의 핵심)

수집한 항목을 아래 셋 중 하나로 분류한다. 등급은 "도구가 붙인 심각도" 가 아니라 **이 프로젝트에서의 실제 영향**으로 정한다.

## 🔴 즉시 수정 필요 — 보안 위험 또는 빌드 실패 수준

- **타입 오류 전부.** 이 프로젝트의 `npm run build` 는 `tsc && vite build` 라, 타입 오류 하나면 **프로덕션 빌드가 실패**한다. 그래서 `tsc --noEmit` 의 에러는 원칙적으로 모두 즉시 수정 대상이다.
- **런타임에 실리는 의존성의 실증적 취약점.** `dependencies`(번들에 포함되어 사용자 브라우저에서 도는 코드)에서 나온 high/critical 이고, 이 앱의 사용 경로와 닿아 있는 것.
- **실제 시크릿 노출.** `src/` 에 진짜 API 키·토큰·비밀번호·비공개 자격증명이 하드코딩되어 커밋되려는 경우. (프런트엔드라도 — 아니 프런트엔드라서 더 — 번들은 공개되므로 치명적이다.)

## 🟡 권장 수정 — 코드 품질·위생 저하 (지금 안 고쳐도 빌드/보안엔 지장 없음)

- **devDependencies 의 취약점.** 빌드·테스트·로컬 서버 전용 도구(예: `json-server`, 번들러 플러그인)에서 나온 것은 사용자에게 배포되지 않는다. 올릴 수 있으면 올리되, 급한 불은 아니다.
- **개선 여지 있는 환경변수 패턴.** 하드코딩된 설정값을 환경변수로 빼면 좋은 경우 (예: 백엔드 URL). 보안 사고는 아니고 위생 문제.

## ⚪ 무시 가능 — false positive 또는 의도된 패턴

- **의도된 개발용 기본값.** 예: `src/api/notes.ts` 의 `const API_URL = 'http://localhost:3001'` 은 JSON Server 로컬 백엔드 주소다. 시크릿이 아니라 개발 기본값이며, 노출돼도 의미 없다.
- **닿지 않는 경로의 취약점.** 이 앱이 실제로 호출하지 않는 의존성 기능의 취약점(예: 쓰지 않는 SSR 경로).
- **고칠 방법이 없는 transitive 취약점.** 상위 패키지가 아직 패치를 안 낸 경우. 무시하되 재평가 시점을 남긴다.

> 애매하면 한 단계 위로 올려 보고하고 개발자에게 판단을 넘긴다. 조용히 무시하는 것보다 "이거 무시해도 될까요?" 가 낫다.

# 파이프라인

```
tdd-refactor 완료 (테스트 Green, 커밋 직전)
        │
        ▼ 1. npx tsc --noEmit      → 타입 오류 수집
        ▼ 2. npm audit             → 의존성 취약점 수집
        ▼ 3. src/ 시크릿·env 노출 grep
        ▼ 4. 🔴/🟡/⚪ 세 등급으로 분류해 보고  [GATE]
        │
        ▼ 5. 승인 후 🔴 "즉시 수정 필요" 항목만 처리
        ▼ 6. 재스캔 → 🔴 0 (클린) 확인
        ▼ (커밋은 사용자가 직접)
```

---

## 단계 1: 타입 오류 수집

```bash
npx tsc --noEmit
```

- 에러를 `파일:라인` 과 메시지로 목록화한다. 출력이 없으면 타입은 클린.
- 여기서 나온 에러는 곧 `npm run build` 실패다(빌드가 `tsc && vite build`). 그래서 기본적으로 🔴.

## 단계 2: 의존성 취약점 수집

```bash
npm audit
```

- 심각도별 개수와, **어떤 패키지가 `dependencies`(런타임)인지 `devDependencies`(개발 전용)인지** 구분해 둔다 — 이 구분이 단계 4 분류의 핵심이다. (필요하면 `npm audit --json` 으로 경로를 정확히 본다.)
- `npm audit fix` 가 semver 호환 범위에서 고칠 수 있는지도 같이 확인하되, **이 단계에서 고치지는 않는다** (단계 5에서 승인 후).

## 단계 3: 시크릿·환경변수 노출 점검

`src/` 에 진짜 비밀값이 하드코딩됐는지, 환경변수를 잘못 노출하는지 본다.

```bash
grep -rniE "api[_-]?key|secret|token|password|passwd|credential|private[_-]?key" src/
grep -rn "import.meta.env\|process.env" src/
```

- 이번 이슈가 바꾼 파일(`git diff main...HEAD --name-only -- src/`)을 **먼저** 본다 — 새로 들어온 시크릿이 가장 위험하다.
- 판단 포인트:
  - 진짜 키/토큰 값이 박혀 있다 → 🔴.
  - 하드코딩된 개발용 URL·포트(예: `localhost:3001`) → ⚪ 또는 🟡(환경변수로 빼면 좋음).
  - `import.meta.env.VITE_*` 사용 → Vite 가 **의도적으로 번들에 공개**하는 값이다. 여기에 비밀을 담았는지만 확인(담았으면 🔴). 단순 사용 자체는 정상.
- 이 프로젝트엔 현재 `.env` 파일이 없다. `.env` 가 생겼다면 그 변수명이 `src/` 에 비공개 값으로 새지 않는지, `.gitignore` 에 `.env` 가 있는지도 함께 확인한다.

## 단계 4: 분류해서 보고 [GATE]

수집한 모든 항목을 `## 분류 기준` 에 따라 🔴/🟡/⚪ 로 나눠 보고한다. **반드시 이 형식**으로:

```
## 보안 점검 결과 — 이슈 N

### 🔴 즉시 수정 필요 (M건)
- [타입] 파일:라인 — 메시지 / 빌드 실패 사유
- [audit] 패키지@버전 (runtime dep, critical) — 영향 경로

### 🟡 권장 수정 (M건)
- [audit] 패키지@버전 (devDependency, high) — 배포 비포함, 업그레이드 권장

### ⚪ 무시 가능 (M건)
- [grep] src/api/notes.ts:3 API_URL=localhost:3001 — 개발 기본값, 시크릿 아님

→ 🔴 항목을 처리할까요? (🟡/⚪ 는 보고만 하고 두겠습니다.)
```

각 항목에 **왜 그 등급인지** 한 줄 근거를 붙인다. 개발자가 승인(또는 일부 선택)할 때까지 아무것도 고치지 않는다. 🔴 가 하나도 없으면 "클린, 커밋해도 됩니다" 로 끝낸다.

## 단계 5: 승인된 🔴 항목만 처리

승인된 "즉시 수정 필요" 만 손댄다. 종류별 처리:

- **타입 오류** — 오류를 낸 코드를 고친다. 그 오류 해소에 필요한 최소 변경만. 테스트 파일은 건드리지 않는다.
- **런타임 의존성 취약점** — `npm audit fix`(semver 호환) 로 시도. `--force` 는 쓰지 않는다. 호환 범위에서 안 고쳐지면 멈추고 "메이저 업그레이드가 필요해 동작 영향 가능 — 어떻게 할까요?" 로 보고.
- **시크릿 노출** — 하드코딩된 비밀값을 제거하고 환경변수/설정으로 옮긴다. 이미 커밋 이력에 들어갔다면 키 회전(rotate)이 필요함을 경고한다(이력 제거는 별도 작업).

판단이 필요한 지점이 나오면(예: 어떤 타입으로 고칠지 모호) 멈추고 묻는다 (원칙 2).

## 단계 6: 재스캔 → 클린 확인

처리한 항목에 해당하는 스캔을 다시 돌려 🔴 가 해소됐는지 확인한다.

```bash
npx tsc --noEmit      # 타입 고쳤으면
npm audit             # 의존성 고쳤으면
```

- 목표는 **🔴 0** (원칙 5). 남은 🟡/⚪ 는 보고서에 그대로 남긴다.
- 처리 과정에서 새 문제가 생기지 않았는지(특히 `npm audit fix` 가 다른 걸 바꿨는지) 확인하고, 가능하면 `npm test` 로 회귀도 본다.
- 마지막에 최종 상태(🔴 0, 남은 🟡/⚪ 목록)를 요약하고, 커밋은 사용자가 한다고 안내한다.

---

## 제약 (반드시 지킬 것)

- **판단 항목은 승인 후에만 수정** — 등급이 갈리거나 고치는 방법이 여럿이면 임의로 정하지 않고 개발자에게 묻는다.
- **테스트 파일 수정 금지** — `*.test.ts(x)` 는 손대지 않는다.
- **`npm audit fix --force` 금지** — 메이저 강제 업그레이드는 동작을 깨뜨린다. `npm audit fix`(호환 범위)까지만.
- **"즉시 수정 필요" 만 자동 처리** — 🟡/⚪ 는 보고만. 커밋 전 게이트의 일은 위험 차단이지 전면 정리가 아니다.
- **점검 범위를 넘는 리팩토링 금지** — 타입 오류를 고치다 구조를 손보고 싶어도 멈춘다. 그건 `tdd-refactor` 단계.

## 분류 예시 (이 프로젝트 기준)

**Input (스캔 결과 일부):**

- `tsc`: `src/components/NoteEditor.tsx:42 - Argument of type 'string | undefined' is not assignable to 'string'`
- `npm audit`: `json-server` 의존 패키지에서 critical 1건
- `grep`: `src/api/notes.ts:3 const API_URL = 'http://localhost:3001'`

**Output (분류):**

- 🔴 **즉시** — `NoteEditor.tsx:42` 타입 오류. `npm run build`(tsc) 가 깨진다.
- 🟡 **권장** — `json-server` 취약점. **devDependency**(로컬 목 서버)라 사용자에게 배포되지 않음. `npm audit fix` 로 호환 범위 업그레이드 시도 권장, 급하진 않음.
- ⚪ **무시** — `API_URL` 하드코딩. JSON Server 개발 기본값이고 시크릿이 아님. (정 신경 쓰이면 🟡 로 올려 환경변수화.)

## 자주 하는 실수 (피할 것)

- **audit 심각도만 보고 등급 매기기**
  - "critical 이니까 즉시" 가 아니다. **런타임 dep 인지 devDependency 인지**, 실제 사용 경로에 닿는지가 등급을 정한다. 학습용 프런트엔드에서 빌드 도구 취약점은 대개 🟡/⚪ 다.

- **`npm audit fix --force` 로 한 방에 해결**
  - 금지다. 메이저가 갈려서 빌드/동작이 깨진다. 호환 범위 `npm audit fix` 까지만, 안 되면 보고.

- **승인 없이 🟡·⚪ 까지 고치기**
  - 커밋 전 게이트는 위험 차단이 목적이다. 권장·무시 항목까지 손대면 변경 범위가 부풀고 커밋이 지저분해진다. 보고만 한다.

- **타입 오류를 `any`·`@ts-ignore` 로 덮기**
  - 빌드는 통과하겠지만 타입 안전성을 버리는 것이다. 진짜 원인을 고친다. 불가피하면 그 이유와 함께 개발자에게 보고.

- **하드코딩 전부를 시크릿으로 오인**
  - `localhost` URL·포트·공개 상수는 시크릿이 아니다. 진짜 비밀값(키·토큰·자격증명)과 구분한다. 반대로 진짜 키를 "설정값" 이라 넘기지도 않는다.

- **재스캔 생략**
  - 고친 뒤 다시 돌려 🔴 0 을 눈으로 확인하지 않으면, 고쳤다고 믿었는데 안 고쳐졌을 수 있다. 처리한 종류의 스캔은 반드시 재실행.
