# 태그 기능 PRD

## 개요

노트에 태그를 추가·삭제하고 노트 상세 화면에서 확인할 수 있는 기능이다.
태그는 노트 객체 내부에 문자열 배열로 저장되며, 저장 버튼 클릭 시 제목·본문과 함께 일괄 전송된다.

---

| --- | ---------- | ------------------------------------------------------------------- | --------------------------------------------------- |
| 1   | 사용자로서 | 노트 편집 화면에서 태그를 입력하고 Enter를 눌러 추가할 수 있다      | 노트를 분류하기 위해                                |
| 2   | 사용자로서 | 추가된 태그를 칩 형태로 본문 아래에서 확인할 수 있다                | 현재 태그 상태를 파악하기 위해                      |
| 3   | 사용자로서 | 태그 칩에 마우스를 올리면 나타나는 x 버튼으로 태그를 삭제할 수 있다 | 잘못 추가한 태그를 제거하기 위해                    |
| 4   | 사용자로서 | 저장 버튼을 눌러야 태그 변경이 서버에 반영된다                      | 제목·본문과 일관된 저장 동작을 기대하기 때문에      |

## 기술 결정

### ADR-01. 컴포넌트 아키텍처: 커스텀 훅 분리

- **Context**: 태그 입력 로직(유효성 검사·정규화·상태 관리)을 어디에 위치시킬지 결정 필요. NoteEditor 인라인, TagInput 제어 컴포넌트, useTagInput 커스텀 훅 분리, Context 액션 네 방향이 검토됨.
- **Decision**: `useTagInput` 훅으로 상태와 로직을 분리하고, `TagInput`은 순수 렌더링만 담당하는 합성 구조 채택.
- **Alternatives**:
  - 인라인: NoteEditor에 로직 직접 작성 → 유효성 로직이 UI에 섞여 단위 테스트 불가
  - TagInput 제어 컴포넌트: 로직이 TagInput 안에 캡슐화되나 훅보다 재사용·테스트가 어려움
  - Context 액션(즉시 저장): spec 불일치, `isCreating` 상태에서 noteId 없어 구조적 충돌 발생
- **Consequences**:
  - 유효성·정규화 로직을 `renderHook`으로 UI 없이 단위 테스트 가능
  - `title`·`content`는 컴포넌트 내 `useState`인데 `tags`만 훅 — 비대칭 발생. 단, 이후 폼 로직을 훅으로 일관화하는 방향의 출발점으로 삼을 수 있음
  - React 순수 API(훅 합성) 활용 패턴을 코드베이스 전반에 정착시키는 기준 사례

### ADR-02. 태그 저장 시점: 저장 버튼 클릭 시 일괄 전송

- **Context**: 태그 추가·삭제 시마다 즉시 서버에 반영할지, 저장 버튼 클릭 시 title·content와 함께 일괄 전송할지.
- **Decision**: 저장 버튼 클릭 시 title·content와 함께 `updateNote`로 일괄 전송.
- **Alternatives**:
  - 즉시 저장(Context 액션): 태그 변경마다 PATCH 호출 → spec 불일치, `isCreating` 시 noteId 없음, 취소 시 롤백 로직 필요, e2e flaky 위험
- **Consequences**:
  - title·content와 동일한 저장 UX — 일관성 확보
  - `isCreating` 케이스에서 노트 생성과 태그가 한 번에 처리되어 자연스럽게 동작
  - 네트워크 요청 최소화
  - 저장 없이 화면을 벗어나거나 새로고침하면 변경한 태그가 모두 소실됨 — 사용자가 저장을 인지하지 못했을 때 작업이 사라질 수 있음

### ADR-03. 유효성 검사 실패 피드백: 조용한 early return

- **Context**: 빈 값, 15자 초과, 10개 초과, 중복 입력 시 사용자에게 어떻게 알릴지.
- **Decision**: `alert` 없이 조용한 early return.
- **Alternatives**:
  - `alert()`: CLAUDE.md 금지 규칙
  - toast 알림: 라이브러리 도입 필요, 이번 스펙 범위 초과
  - 인라인 에러 메시지: 구현 복잡도 증가, 디자인 결정 필요
- **Consequences**:
  - 기존 에러 처리 규칙(CLAUDE.md)과 일관성 유지
  - 사용자가 왜 추가가 안 되는지 파악하기 어려움 → 향후 인라인 피드백으로 개선 여지

### ADR-04. 정규화 전략: trim + toLowerCase

- **Context**: 동일한 태그를 `"React"`, `"react"`, `" react "` 등 다양한 형태로 입력할 수 있어 중복 감지 기준 정의 필요.
- **Decision**: 저장 전 `trim() + toLowerCase()` 적용 후 중복 비교.
- **Alternatives**:
  - 원본 케이스 유지: `"React"`와 `"react"`가 별개 태그로 취급됨
  - 서버 측 정규화: 클라이언트 즉시 반영 불가, API 레이어 변경 필요
- **Consequences**:
  - 사용자가 입력한 케이스와 저장된 값이 다를 수 있음 (예: `"TypeScript"` → `"typescript"`)
  - 일관된 중복 감지 보장
  - 향후 필터링·자동완성에서도 동일 정규화 기준 재사용 가능

## Out of Scope

이번 스코프에서 명시적으로 제외하는 항목이다.
아래 항목을 이번 이터레이션에서 구현하거나 대비한 설계를 미리 넣지 않는다.

### 검색/필터

- 태그 기반 노트 검색 UI 및 API
- 다중 태그 AND/OR 필터링
- 태그 클릭 시 해당 태그가 달린 노트만 보여주는 뷰
- 제목/본문/태그를 결합한 통합 검색
- URL 쿼리 파라미터를 통한 필터 상태 공유

### 전역 태그 관리

- 전체 태그 목록(vocabulary)을 조회/관리하는 별도 화면
- 태그 이름 일괄 변경(rename) 및 병합(merge)
- 태그 삭제 시 모든 노트에서 동시 제거(전역 카스케이드 삭제)
- 사용 중인 태그 수 집계/통계(most used, unused 등)

### 자동완성 및 추천

- 입력 중 기존 태그 목록 기반 autocomplete drop-down
- 본문/제목을 분석한 태그 자동 제안(AI 추천 포함)
- 최근 사용 태그 빠른 선택 칩

### 데이터 정규화 및 마이그레이션

- 별도 `tags` 컬렉션/테이블로의 정규화
- 태그 ID 부여 및 참조 기반 저장 구조
- 기존 노트를 대상으로 한 태그 데이터 이관 스크립트(빈 배열 기본값 외)

### 고급 UI/UX

- 태그별 색상 지정, 아이콘, 이모지 첨부
- 드래그 앤 드롭으로 칩 순서 변경
- 태그 그룹/카테고리(계층 구조, 부모-자식 관계)
- 가나다·알파벳 순 태그 정렬
- 노트 목록(`NoteItem`)에서의 태그 뱃지 표시
- 키보드로 칩 포커스 이동 및 방향키 탐색
  (현재 스코프는 Enter / Backspace만)

### 권한/협업/동기화

- 사용자별/팀별 태그 권한 분리
- 실시간 협업 편집 시 태그 충돌 해결(CRDT/OT)
- 오프라인 편집 후 태그 머지 전략

### 검증 정책 확장

- 태그 문자 집합 제한(특수문자/이모지 금지 등)
  현재는 길이(15자)·개수(10개)·공백·중복만 검증한다.
- 금칙어/필터링 정책
- 다국어 정규화(NFC/NFD), 유사어 처리

### 성능/관측

- 태그 인덱싱, 캐싱 전략
- 태그 변경 이력(audit log) 및 분석 이벤트 수집

### 테스트 범위 외

- E2E 시나리오(태그 기반 검색 흐름 등)는 검색 기능 도입 시 다룬다.

## 용어 정의

| 용어                   | 정의                                                                                                                     |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| 태그(Tag)              | 노트를 분류하기 위해 붙이는 짧은 문자열 레이블. 노트 객체 내 `tags` 배열에 저장된다.                                     |
| 칩(Chip)               | 확정된 태그 1개를 시각적으로 표현하는 pill 형태의 UI 요소. hover 시 × 버튼이 노출된다.                                   |
| Chip Input             | 텍스트 input과 확정된 칩 목록을 한 컨테이너에 결합한 입력 컴포넌트. Enter로 확정, Backspace로 마지막 칩 삭제를 지원한다. |
| 확정(Commit)           | 입력 중인 텍스트를 칩으로 변환해 `tags` 배열에 추가하는 동작. 트리거는 Enter 키.                                         |
| 정규화(Normalization)  | 저장 전 `trim() + toLowerCase()`를 적용하여 태그 값을 통일하는 처리. 예: `" React "` → `"react"`                         |
| trim                   | 입력값 양옆의 공백 제거. trim 결과가 빈 문자열이면 확정하지 않는다.                                                      |
| 중복(Duplicate)        | 정규화 후 기존에 추가된 태그와 동일한 값. 추가 시 무시된다.                                                              |
| 최대 길이              | 태그 1개당 허용되는 최대 문자 수. 15자.                                                                                  |
| 최대 개수              | 노트 1개당 허용되는 최대 태그 수. 10개.                                                                                  |
| Placeholder            | 태그가 0개일 때 input에 표시되는 안내 문구 (`태그 입력 후 Enter`).                                                       |
| NoteEditor             | 노트 생성/편집 폼 컴포넌트 (`src/components/NoteEditor.tsx`). 태그 UI가 배치되는 위치.                                   |
| updateNote             | 노트 수정 API 함수 (`src/api/notes.ts`). 태그는 이 호출에 포함되어 저장된다.                                             |
| 로컬 상태(Local State) | 저장 버튼을 누르기 전까지 컴포넌트 내부에만 존재하는 임시 상태. 화면을 벗어나면 사라진다.                                |
| 일괄 저장(Batch Save)  | 태그 변경을 즉시 서버에 보내지 않고, 저장 버튼 클릭 시 제목·본문과 함께 한 번에 `updateNote`로 전송하는 방식.            |
| Early Return           | 유효하지 않은 입력(빈 문자열, 글자 수 초과, 개수 초과, 중복)에 대해 alert 없이 조용히 함수를 종료하는 처리 방식.         |
